# izLearn — Regulatory Compliance Mapping

This document maps each required clause of **21 CFR Part 11**, **EU Annex 11** and **ALCOA++** to the
concrete mechanism in the izLearn codebase that satisfies it.

---

## 21 CFR Part 11 — Subpart B (Electronic Records)

### §11.10(a) — Validation of systems to ensure accuracy, reliability, consistent performance
The system is built on typed, schema-validated layers: every request body is validated against shared
**Zod** schemas (`shared/schemas`) used identically on client and server, and the database schema is
applied through **explicit, version-controlled Prisma migrations** (`backend/prisma/migrations`, never
`db push`). A boot-time health check (`verifyAuditTrigger`, surfaced at `GET /api/health`) confirms the
audit-immutability triggers exist before the system is considered operational, and an automated test
suite (`backend/src/tests`) exercises the critical compliance logic (audit immutability, RBAC, scoring,
file security, password policy).

### §11.10(b) — Ability to generate accurate and complete copies of records
All 12 reports and the audit trail can be exported as **PDF (Puppeteer), CSV and XLSX (ExcelJS)** with
full header/footer provenance (org, reference number, generated-by, page X of Y) via
`report.service.ts` and `auditTrail.controller.ts`. Complete database copies are produced by the
`pg_dump`-based backup service (`backup.service.ts`) with SHA-256 checksums for integrity.

### §11.10(c) — Protection of records for accurate, ready retrieval throughout the retention period
**No record is ever hard-deleted** — every "delete" sets `isDeleted = true` (soft delete), preserving
history; the `AuditTrail` and `ElectronicSignature` tables are insert-only and protected by PostgreSQL
triggers. Records are indexed for retrieval, retained indefinitely, and protected by scheduled,
checksum-verified backups (RPO 24h; see Annex 11 §9).

### §11.10(d) — Limiting system access to authorized individuals
Access requires a valid JWT issued only after bcrypt/LDAP authentication (`auth.middleware.ts`); every
route is guarded by `authenticate` plus role-based `requirePermission`. Additional controls: single
active session per user (Redis-backed), failed-login lockout, inactivity lock, and password policy/expiry
/history enforcement.

### §11.10(e) — Secure, computer-generated, time-stamped audit trails
A Prisma `$use` middleware (`auditTrail.middleware.ts`) intercepts **every** create/update/delete on GMP
models and writes an `AuditTrail` row **inside the same database transaction** — if the audit insert
fails, the data change rolls back. Timestamps are generated by the **database server** (`DEFAULT NOW()`),
the trail is **insert-only** (enforced by a PostgreSQL trigger that raises on UPDATE/DELETE), and the
middleware is registered unconditionally at startup with no flag to disable it.

### §11.10(f) — Operational system checks (enforced sequencing of steps)
Workflows are enforced in the service layer: the JD lifecycle (`DRAFT → UNDER_REVIEW → APPROVED`) rejects
out-of-order transitions; TNI approval is the mandatory gate that activates training assignments;
assessment attempts are blocked at `maxAttempts`; a training schedule rejects a trainer who is also a
trainee (HTTP 400); and no event date may be in the future (shared `pastOrPresentDate` refinement +
server-time defaults).

### §11.10(g) — Authority checks
`requirePermission(module, action)` (`rbac.middleware.ts`) loads **all** of a user's active roles and
grants access only if at least one role's permission matrix allows the requested module + action. Denied
attempts are recorded in the audit trail as `ACCESS_DENIED`. Roles and their matrices are managed under
Master Setup with e-signed changes.

### §11.10(h) — Device (terminal) checks
On login the device fingerprint (`deviceInfo`) is captured and stored with the session in Redis and
`UserSession`. Single-session enforcement detects an existing session on another device and requires the
user to explicitly terminate it (`SESSION_TERMINATED` audited); orphaned/browser-closed sessions are
detected and resolved before a new sign-in proceeds.

### §11.10(i) — Education, training and experience of personnel
izLearn **is** the system of record for personnel training, competency evaluation and job descriptions,
providing the documented evidence that individuals are trained for their roles. Operationally, project
roles and responsibilities for those who build/maintain the system are documented in the SOPs that
accompany this deployment.

## 21 CFR Part 11 — Subpart C (Electronic Signatures)

### §11.50 — Signature manifestations
Each `ElectronicSignature` record captures the signer's **printed full name** (snapshot), the
**server-generated date/time**, and the **meaning** of the signature (controlled vocabulary: Approved /
Reviewed / Rejected / Performed / Acknowledged). This block is displayed inline wherever a signed record
is viewed (`SignatureBlock` component): "Signed by [Name] on [DD/MM/YYYY HH:MM] — Meaning: [Approved]".

### §11.70 — Signature/record linking
Signatures are applied with a **two-component** challenge (re-entered username + a separate signature
password, verified server-side in `eSignature.service.ts`). The resulting `ElectronicSignature.id` is
stored as a foreign key on the signed record (e.g. `JobDescription.signatureId`, `TNI.signatureId`), the
signature table is immutable, so a signature can never be copied, transferred or detached from its record.

## EU Annex 11

### §9 — Data storage & backup (incl. RPO/RTO)
Data is stored in PostgreSQL with PII protection available via `pgcrypto`. Backups run on a configurable
`pg_dump` cron (`db-backup` Bull job) writing a dump + SHA-256 checksum to a configured destination;
`scripts/verify-backup.sh` validates integrity and `RESTORE.md` documents recovery.
**Disaster recovery targets: RPO = 24 hours** (daily automated backup) **and RTO = 4 hours** (restore +
verify + service restart per `RESTORE.md`).

### §10 — Audit trails
Satisfied by the same always-on, same-transaction, server-timestamped, immutable audit trail described in
§11.10(e). A dedicated, filterable Audit Trail screen (date range, user, action, entity) with e-signed
PDF/CSV/XLSX export is provided for review.

## ALCOA++

- **Attributable** — every record carries `createdBy`; every audit/signature row carries the actor's id
  and full-name snapshot, captured via request-scoped `AsyncLocalStorage`.
- **Legible** — records are structured/typed and exportable to human-readable PDF/CSV/XLSX.
- **Contemporaneous** — all authoritative timestamps come from the DB server at the moment of the action;
  client timestamps are never trusted, and future dates are rejected.
- **Original** — the first capture is preserved; updates create new audit snapshots (old + new values)
  rather than overwriting history.
- **Accurate** — shared Zod validation, enforced workflows, and same-transaction auditing keep records and
  their trail consistent.
- **Complete** — soft-delete retains all records; the audit trail captures create/update/delete plus
  login/logout/export/print/e-sign/config/permission events.
- **Consistent** — server-generated, timezone-aware timestamps and a single shared validation layer.
- **Enduring** — immutable audit/signature tables, indefinite retention, and checksum-verified backups.
- **Available** — indexed retrieval, reporting, and the always-on `GET /api/health` readiness check.

## Attack surface (penetration-testing readiness)

| Surface | Control |
|---|---|
| `POST /api/auth/*` | Rate limited 10/min/IP (audited `RATE_LIMITED`), bcrypt(12), lockout, no copy-paste on login |
| All `/api/*` (except `/health`, `/auth/login`, `/auth/refresh`) | JWT auth + RBAC authority checks |
| File uploads | Extension + MIME match, double-extension rejection, size cap, UUID storage names, AV scan hook |
| Rendered HTML (JD, announcements) | DOMPurify sanitisation (stored-XSS defence) + React escaping |
| SQL | Prisma parameterised queries only — no raw string interpolation |
| Transport | Helmet (HSTS, strict CSP), CORS whitelist from `SystemConfig` |
| Records | Insert-only audit & signature tables enforced at the DB layer |
| Dependencies | `npm run security:audit` (`npm audit --audit-level=high`) |
| Observability | `GET /api/health` (DB/Redis/trigger), Swagger at `/api-docs` |
